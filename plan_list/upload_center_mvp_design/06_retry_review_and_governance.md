# Phase 6: 重试、人工审核与治理补强

## 目标

完善失败处理、重试和人工审核链路，让 Upload Center 具备基本运维能力和可持续使用能力。

## 前置条件

- 上传主链路已能完成“上传 -> 抽取 -> 报告 -> sync”。
- 前端已能查看上传任务详情。

## 任务清单

- 实现 `POST /api/uploads/{upload_id}/retry`。
- 约束重试规则：
  - 仅 `failed` 或 `needs_review` 可重试
  - 重试增加 `retry_count`
  - 默认生成新的 `report_id`
- 明确 `needs_review` 进入条件：
  - 抽取文本过短
  - 标题缺失或异常
  - 格式识别异常
  - AI 输出质量不足
- 在详情页增加人工审核提示。
- 保留失败诊断产物：
  - `error_log`
  - `model_output`
  - `report_preview`
- 评估是否在 `Admin` 或 `Tasks` 中加入上传异常概览入口。

## 交付物

- Retry API
- `needs_review` 判定逻辑
- 失败诊断与审核提示能力
- 基础治理视图或入口补强

## 验收标准

- 失败任务可被显式重试，且 `retry_count` 会递增。
- 重试不会静默覆盖旧报告文件。
- `needs_review` 任务可在前端被明显识别。
- 失败原因和关键诊断产物可被查看。
- 高频失败类型可以从日志或列表中被定位。

## 风险与注意事项

- 重试策略从一开始就要避免“覆盖原报告”，否则审计链会混乱。
- 人工审核入口先做轻量提示，不急于做复杂审批流。
