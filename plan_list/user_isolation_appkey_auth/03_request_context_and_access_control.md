# Phase 3: 请求上下文与访问控制
## 目标

让所有业务接口在后端可信上下文下运行，把身份、工作区和访问边界统一收敛到依赖层与 Service 层，彻底移除匿名全局访问模式。

## 前置条件

- 已具备稳定的登录态和 `UserContext`。
- 数据模型已支持 `workspace_id`。
- 路径解析已具备按工作区落根能力。

## 任务清单

- 设计统一鉴权依赖：
  - `get_current_user`
  - `get_current_workspace`
- 为业务路由统一接入登录校验。
- 将 Service 层方法签名改为显式接收 `ctx` 或 `workspace_id`。
- 统一规范 SQL 查询必须追加 `workspace_id` 条件。
- 统一规范详情接口的跨工作区访问行为：
  - 推荐返回 `404`
- 为文件读取与原文接口增加二次归属校验。
- 明确角色差异：
  - `user`
  - `admin`
- 明确哪些接口首版仍应限制为普通工作区视角：
  - `reports`
  - `uploads`
  - `search`
  - `wiki`
  - `query`
  - `tasks`
  - `conflicts`
  - `sync`

## 交付物

- 接口鉴权矩阵。
- Router 依赖接入规范。
- Service 层上下文传递规范。
- 404 / 401 / 403 语义说明。

## 验收标准

- 未登录请求无法访问业务接口。
- 所有列表和详情接口都默认按当前工作区过滤。
- 即使知道其他工作区资源 ID，也无法读取数据。
- 文件读取、原文预览和详情接口的隔离行为一致。

## 风险与注意事项

- 不能只在 Router 层做过滤，Service 与文件访问必须同时兜底。
- 若存在遗漏的内部调用链路，可能绕过 Router 直接访问全局数据。
