# Phase 5: 前端登录与会话接入
## 目标

将前端从匿名全局访问模式切换为“APPKEY 登录后按工作区访问”，补齐登录页、会话态、当前用户展示与未登录跳转逻辑。

## 前置条件

- 登录接口与 `GET /api/auth/me` 已定义。
- 后端已支持基于会话态解析当前用户。
- 业务接口已改为默认要求登录。
- 后端登录逻辑已确定为直接调用：
  - `GET https://sg-al-cwork-web.mediportal.com.com/user/login/appkey`
  - query 传入 `APPKEY`

## 任务清单

- 新增 APPKEY 登录页。
- 设计登录表单交互：
  - 输入 APPKEY
  - 登录中状态
  - 错误提示
- 明确前端只调用本系统登录入口，不直接在前端暴露外部登录接口。
- 登录成功后接入会话存储：
  - Cookie 或 Token
- 新增当前用户 / 当前工作区显示区域。
- 统一 API 请求封装，自动带登录态。
- 统一处理 `401`：
  - 清理前端会话状态
  - 跳回登录页
- 设计登出行为：
  - 调用 `POST /api/auth/logout`
  - 清理页面状态
- 评估现有 `Admin` 页面是否需要区分：
  - 普通工作区视角
  - 平台管理视角

## 交付物

- 登录页结构说明。
- 前端请求与会话封装规范。
- 未登录、会话过期、登出后的页面流转说明。

## 验收标准

- 未登录无法进入业务页。
- 登录成功后可正常访问当前工作区数据。
- 会话过期后页面会自动回到登录页。
- 页面能够显示当前用户和工作区信息。

## 风险与注意事项

- 不建议前端把 `APPKEY` 长期存储在 localStorage。
- 若前端状态清理不完整，可能出现页面残留旧工作区数据的假象。
