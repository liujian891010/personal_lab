# Phase 6: 历史数据迁移、回填与切换
## 目标

将当前全局共享数据平滑迁移到工作区模型，保证历史文件、数据库记录和索引之间的一致性，并提供可执行的切换与回滚方案。

## 前置条件

- 新 schema 与工作区目录结构已定稿。
- 后端新旧代码兼容路径已明确。
- 已确认历史数据归属策略。

## 任务清单

- 定义历史工作区映射策略：
  - 单工作区兜底：`legacy-default`
  - 或按用户批量映射
- 编写数据回填策略：
  - 为历史表记录补 `workspace_id`
  - 必要时补 `created_by / updated_by`
- 编写文件迁移策略：
  - 全局 `reports/` -> 工作区 `reports/`
  - 全局 `knowledge/` -> 工作区 `knowledge/`
  - 全局 `uploads/` / `raw/` -> 工作区对应目录
- 重写 `file_path / storage_path / slug` 等路径字段。
- 重建 `reports / wiki / FTS` 索引。
- 设计切换 runbook：
  - 发布前检查
  - 只读窗口
  - 执行迁移
  - 重建索引
  - 核验
  - 切换流量
- 设计回滚 runbook：
  - 恢复数据库备份
  - 恢复目录快照
  - 回切旧版本

## 交付物

- 迁移脚本设计说明。
- 数据回填规则清单。
- 上线切换 runbook。
- 回滚 runbook。

## 验收标准

- 历史数据迁移后仍可在所属工作区正常访问。
- 文件、数据库和索引之间不存在孤儿记录或失配路径。
- 切换失败时可回滚到迁移前状态。
- 迁移过程和结果可重复验证。

## 风险与注意事项

- 目录迁移和 DB 回填必须按同一批次校验，否则极易出现路径失配。
- 不建议在持续写入状态下执行无窗口迁移。
- 回滚方案必须在正式切换前实际演练。
